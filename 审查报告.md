## 一、结论先行

你现在对概念的理解是正确的：

1. TCP 交付的是有序字节流，不交付“消息”。
2. 一次 `Receive` 得到的是“本次收到的一段字节”，它可能不足一条消息，也可能包含多条消息。
3. 分帧器根据协议头从字节流中恢复完整帧边界。
4. 完整帧仍然是编码后的字节；消息解析器再把帧中的类型与消息体解析成 `StringMsg`、`PlayerMsg` 等消息对象。

当前技术债的根因不是你不理解分包/粘包，而是以下职责被揉进了少数几个类：

- Socket 连接和关闭；
- 线程与任务生命周期；
- TCP 字节流累积和分帧；
- 消息序列化；
- 消息类型分发；
- Unity 主线程消费；
- 服务端客户端集合管理。

因此建议保留现在的简单协议，但先把边界拆清楚。最合适的目标链路是：

```text
消息对象
  ↓ MessageCodec（序列化消息体、写协议头）
完整帧字节：Header + Payload
  ↓ TCP Transport（保证把全部字节发完）
TCP 字节流
  ↓ TcpFrameDecoder（累积字节并恢复完整帧）
NetworkFrame：TypeId + Payload
  ↓ MessageCodec / MessageRegistry
消息对象
  ↓ 主线程或服务端业务分发器
业务逻辑
```

不要推倒重写整个项目。先修 P0，再抽出纯 C# 的 `TcpFrameDecoder`，现有代码就能逐步变成以后可继续学习和扩展的底座。

## 二、统一术语与建议命名

### 2.1 建议固定的术语

| 术语 | 含义 | 代码中的合适类型 |
| --- | --- | --- |
| 字节（byte） | 传输和编码处理的最小单位 | `byte` |
| 接收块（receive chunk） | 一次 `Receive` 实际取到的那段字节，不带消息边界语义 | `buffer + bytesRead` 或 `ReadOnlySpan<byte>` |
| 流累积区（stream accumulator） | 保存尚未组成完整帧的 TCP 字节 | 解码器内部缓冲区 |
| 协议头（header） | 当前协议中的 `MessageTypeId + PayloadLength`，共 8 字节 | `FrameHeader` 或常量定义 |
| 消息体（payload/body） | 协议头之后的编码数据 | `byte[] payload` |
| 完整帧（frame） | 一条完整的线上数据单元，即 `Header + Payload` | `byte[] frameBytes` 或 `NetworkFrame` |
| 消息（message） | 已解析、具有业务含义的对象 | `StringMsg`、`PlayerMsg` |

当前 `_processedMsgQueue` 里装的是 `byte[]`，严格说它是“完整帧队列”，还不是“消息对象队列”。这正是现有命名最容易再次混淆的地方。

### 2.2 旧名称到新名称的建议映射

| 当前名称 | 建议名称 | 原因 |
| --- | --- | --- |
| `originalBytesBucket` / `buffer` | `_receiveBuffer` | 它只是每次调用 `Receive` 使用的临时接收缓冲区 |
| `_originalBytesQueue` | `_streamAccumulator`，并封装进 `TcpFrameDecoder` | 它保存的是尚未完成分帧的 TCP 流数据 |
| `_processedMsgQueue` | `_completeFrameQueue` | 元素仍是完整帧字节，不是消息对象 |
| `_pendingSendBytesQueue` | `_outgoingFrameQueue` | 队列中的元素应当是一条待发送的完整帧 |
| `needSendBytesInQueue` | 局部变量 `frameBytes` | 不需要成为跨循环共享字段 |
| `msgBodyCount` / `MsgBodyCount` | `payloadLength` / `PayloadLength` | 这里表示字节长度，不是元素数量 |
| `ProcessTCPOriginalBytes` | `Append` + `TryReadFrame` | 分帧器应该表达“追加流数据”和“尝试读完整帧” |
| `Progress...` | `Process...`、`SendLoop`、`ReceiveLoop` | `Progress` 是“进度”，不是“处理” |
| `DeSerializeFormBytes` | `DeserializeFromBytes` | 修正拼写和介词 |
| `GetAllBytesLength` | `GetEncodedLength`，或直接删除 | “All” 没有说明到底是消息体还是完整帧 |
| `allBytes` | `_encodedBuffer` | 说明这是编码/解码所用缓冲区 |

注释应主要说明“不明显的协议约束和设计原因”，不要逐行重复代码。比如“长度字段是消息体长度，不含 8 字节头”值得写；“游标加 4”无需反复写。

## 四、关键问题与修改思路

### 4.1 P0：`PlayerMsg` 目前不可序列化

调用链如下：

```text
PlayerMsg.SerializeToBytes()
→ InitAllBytes()
→ PlayerMsg.GetAllBytesLength()
→ GetMsgHeaderCount()
→ TrySetMsgTypeID()
→ SetMsgBodyCount()
→ throw new NotImplementedException()
```

所以客户端“发送自定义类型”按钮和服务端 `SendPlayerMsg` 命令都会在第一次序列化时失败。

当前协议下，`PlayerMsg` 的消息体字节长度至少应当是：

```text
playerId 的 4 字节
+ PlayerData 长度前缀的 4 字节
+ PlayerData 自身编码后的字节长度
```

**短期止血**：两端同时正确实现该长度。  
**推荐根治**：不要再让每个消息子类设置和缓存消息体长度。先序列化 payload，得到 `payload.Length` 后由统一 `MessageCodec` 写协议头。这样 `PlayerMsg` 不可能忘记实现长度，也不会因字段改变而写入旧长度。

### 4.2 P0：存在两处确定的 CPU 空转

客户端发送任务在队列为空时仍执行：

```csharp
while (_needConnectToServer)
{
    if (_pendingSendBytesQueue.Count > 0)
    {
        // send
    }
}
```

服务端 `ProcessAllClientMsg` 无论有没有客户端、有没有数据，都持续遍历并查询 `Available`。这两处都没有阻塞等待、信号或延迟，会尽可能占满一个 CPU 核心。

不建议用 `Thread.Sleep(1)` 掩盖问题；它只是把持续空转改成定时轮询。

**适合你当前学习阶段的改法**：

- 客户端发送队列使用会等待数据的生产者/消费者容器，例如 `BlockingCollection<byte[]>`；有数据时唤醒发送线程，无数据时线程休眠。
- 服务端每个 `ClientSession` 使用自己的阻塞式 `Receive` 循环；当前是学习项目，少量客户端时“一连接一个接收循环”足够清楚。
- 等学到真正异步后，把阻塞式 `Accept/Receive/Send` 换成对应的 Async API；帧解码器和消息编解码器不需要跟着重写。

如果现在就使用异步模型，则应使用真正的 `AcceptAsync/ReceiveAsync/SendAsync` 和 `CancellationToken`，而不是把同步阻塞方法和轮询循环整体包进 `Task.Run`。

### 4.3 P0：集合存在数据竞争和顺序风险

#### Unity 客户端

- 主线程在 `SendBytesToServer` 中 `Enqueue`，线程池线程在发送循环中读取 `Count` 并 `Dequeue`。
- 接收线程向 `_processedMsgQueue` 入队，Unity 主线程在 `Update` 中通过“先判断、再出队”消费。
- `Queue<T>` 不保证这些并发操作安全；`Count > 0` 与 `Dequeue()` 也不是一个原子操作。

建议：

- 待发送帧：`BlockingCollection<byte[]>`，或“线程安全队列 + 可等待信号”。
- 已完成帧：`ConcurrentQueue<NetworkFrame>`，只暴露一个 `TryDequeueCompleteFrame(out frame)`，删除 `IsHave...` 与 `Get...` 两步接口。
- 每个 Socket 只允许一个发送者，才能同时保证线程安全和发送顺序。

#### 服务端

- Accept 线程在字典中 `Add`，轮询线程和控制台线程同时枚举该字典，关闭时又会 `Clear`。
- `ConnectClientSocket` 每恢复一条帧就启动一个 `Task`；多个任务并发从普通队列 `Dequeue`，业务处理顺序也不再有保证。
- 已关闭连接没有从服务端字典移除，之后只会成为永久空轮询项。

建议：

- 使用 `ConcurrentDictionary<int, ClientSession>`，或用同一个锁保护增加、移除和快照枚举。
- `ClientSession` 关闭时回调服务器移除自己。
- 一个连接的已完成帧按接收顺序串行分发；不要“一条消息一个 Task”。需要并行的业务以后在业务层明确选择，而不是在底层隐式打乱顺序。
- 广播先取得活动连接快照，再逐个发送；单个连接失败不应中断其他连接。

### 4.4 P0：`Socket.Send` 不保证一次发完参数中的全部字节

两端都调用了 `Socket.Send(frameBytes)`，但忽略返回的实际发送字节数。TCP 保证顺序，不保证一次 `Send` 接收整个数组，也不保证对端一次 `Receive` 对应一次 `Send`。

应提取统一的 `SendAll` 逻辑：

```text
offset = 0
while offset < frameBytes.Length:
    sent = Socket.Send(frameBytes, offset, remaining, ...)
    sent <= 0 时视为连接失败
    offset += sent
```

异步版本也要循环调用 `SendAsync`，直到整个帧全部发送。每个连接只能有一个发送循环，多个业务线程只负责入队。

### 4.5 P0：网络长度字段必须视为不可信输入

当前分帧器直接把第 5～8 字节解释为 `msgBodyCount`，随后进行游标计算、数组分配和出队，没有检查：

- 是否为负数；
- 是否超过允许的最大消息体；
- `8 + payloadLength` 是否整数溢出；
- 完整帧实际长度是否与头中长度一致。

反序列化字符串和自定义类型时也直接信任内部长度，可能发生越界、负长度分配或过大分配。

建议先集中定义：

```text
HeaderSize = 8
MaxPayloadLength = 1 MiB（初始值可按项目实际调整）
ByteOrder = LittleEndian（或者 BigEndian，但必须固定）
```

分帧时顺序必须是：

1. 可用字节不足 8：保留并等待。
2. 读取 `typeId` 和 `payloadLength`。
3. 检查 `0 <= payloadLength <= MaxPayloadLength`。
4. 用受检查的加法计算 `frameLength = HeaderSize + payloadLength`。
5. 可用字节不足 `frameLength`：保留并等待。
6. 否则产出一条完整帧并继续尝试下一条。

非法长度通常应该记录原因并关闭当前 TCP 连接。因为没有魔数和重同步机制时，继续在已失去边界的字节流上猜测下一帧是不可靠的。

消息体解析应使用带 `Remaining` 检查的 `ByteReader`：每次读 `int`、字符串或嵌套对象前先验证剩余长度；字符串/嵌套对象也要有独立上限；完成后验证没有意外剩余字节。

### 4.6 P1：当前异步方法不可管理

`ProgressSendBytesAsync`、`ProgressReceivedBytesAsync`、`AutoAcceptSocket`、`ProcessAllClientMsg` 和 `ProgressReceivedMsg` 全是 `async void`。除事件处理器外，`async void` 的异常无法被调用者正常 `await` 和集中处理，也不能在关闭阶段等待任务结束。

建议：

- 循环方法返回 `Task`，字段保存 `_sendLoopTask`、`_receiveLoopTask`、`_acceptLoopTask`。
- 使用明确的连接状态和 `CancellationTokenSource`，而不是多个线程共享普通 `bool`。
- 关闭流程先只执行一次状态迁移，再停止接收新发送项，关闭/释放 Socket 以唤醒阻塞调用，最后等待循环退出。
- 预期中的“因为主动关闭 Socket 而产生的异常”与真正网络异常分开记录。
- `Close` / `Dispose` 必须幂等：重复调用不会再次 `Shutdown` 空对象，也不会抛出未处理异常。

如果当前暂时采用专用阻塞线程，也应保存线程/任务句柄，并在关闭时等待它们退出，而不是发起后完全遗忘。

### 4.7 P1：服务端断开检测和连接清理不完整

客户端接收循环正确检查了 `Receive == 0`。服务端却先判断 `Available <= 0` 就返回，因此当客户端正常断开且没有剩余数据时，服务端可能永远不会调用 `Receive`，也就看不到返回值 0。

修改方向：

- 移除基于 `Available` 的持续轮询。
- 每个会话直接阻塞在 `Receive`；返回 0 时进入统一关闭流程。
- 任何接收、发送或解析失败都只关闭当前会话，并通知 `ListenSocket` 从活动连接表移除。
- 会话的 Socket 字段关闭后不要继续保留在字典中。

这也是以后心跳的基础：心跳是发现“连接表面存在但对端实际上失联”的机制，不能用来弥补本来就没处理的正常 TCP 关闭。

### 4.8 P1：消息长度设计存在隐藏调用顺序和陈旧缓存

`MsgBase` 用两个特殊负数表示“尚未初始化”，再通过 `TrySetMsgTypeID` 同时触发类型和消息体长度计算。`StringMsg` 又依赖 `GetAllBytesLength` 先更新 `cacheMsgBodyCount`。这造成：

- 子类必须知道基类内部调用顺序；
- `PlayerMsg` 很容易遗漏实现；
- 同一个 `StringMsg` 第一次序列化后若修改 `msg` 再序列化，缓存虽然更新，但 `MsgBodyCount` 已不再是哨兵，协议头可能仍写旧长度；
- 从网络读到的 `MsgTypeID` 和 `MsgBodyCount` 被写回消息对象，协议元数据与业务数据混在一起；
- 两个公开 setter 允许外部制造不一致状态。

推荐结构：

- 消息类型用只读属性，例如 `MessageTypeId`，不使用魔法哨兵和 setter。
- 消息类只负责 payload 的字段读写，不负责协议头。
- `MessageCodec.Encode(message)` 先得到 payload 字节，再用真实的 `payload.Length` 写头。
- `MessageCodec.Decode(frame)` 根据帧中 `typeId` 从注册表创建消息并解析 payload。

如果暂时不准备改这么大，最低限度也要改为每次序列化都重新计算 `GetPayloadLength()`，由基类在同一次序列化调用中把该值同时用于数组分配和头部写入，删除缓存和负数哨兵。

### 4.9 P1：当前分帧算法正确方向里夹着不必要的分支和分配

现有代码能表达“头不足就等、体不足就等、有完整帧就取出”的核心思想，但无需区分三个独立分支：粘包、刚好一包、分包。统一条件即可：

```text
available < HeaderSize       → 等待更多字节
available < completeLength   → 等待更多字节
否则                         → 取出一条完整帧，继续循环
```

只要取出一帧后继续循环，自然就处理了所谓“粘包”；不需要把“粘包”当成异常状态。

性能方面：

- `Queue<byte>.ToArray()` 每次看协议头都复制整个剩余队列。
- 每个接收块逐字节 `Enqueue`，每个完整帧又逐字节 `Dequeue`。
- 客户端固定 1 MiB 接收数组，服务端每个连接也固定 1 MiB，但一次接收缓冲区不需要等于最大消息长度。

建议新建纯 C# `TcpFrameDecoder`，内部用连续 `byte[]` 加读写下标：

- `Append(receiveBuffer, bytesRead)` 把本次接收块追加到未消费区；
- `TryReadFrame(out frame)` 只在完整时推进读取位置；
- 空间不足时先压缩已消费区域，再按上限扩容；
- 接收临时数组先用 8 KiB～16 KiB 即可，跨多次接收的数据由解码器累积；
- 第一版先保证正确和有限内存，再考虑 `ArrayPool<byte>` 等优化。

### 4.10 P1：两端协议代码重复，必须建立单一真相来源

客户端和服务端各自复制了：

- 基础二进制读写；
- 消息头；
- 消息类型 ID；
- `StringMsg`、`PlayerMsg`、`PlayerData`；
- 分帧逻辑。

任何一侧改了字段顺序、长度含义或类型 ID，另一侧漏改都会产生难排查的协议错误。

推荐把纯 C# 协议代码放在 Unity 工程内一个不引用 `UnityEngine` 的目录，例如：

```text
Assets/_Project/Scripts/Networking/Protocol/
```

Unity 直接编译这里的源码；服务端 `.csproj` 用 `<Compile Include="...">` 链接同一批物理源码，而不是再复制一份。也可以以后独立成 `netstandard` 类库，但现阶段“同一份源码被两端引用”更容易理解和维护。

建议给共享代码加命名空间，例如 `MyNetwork.Protocol`。Socket、MonoBehaviour、Console 等平台专属代码不能放入共享协议层。

### 4.11 P1：现有分包/粘包测试方法不稳定

`Send` 调用边界不是 TCP 接收边界。即使连续发送两个数组，TCP 仍可拆开；即使间隔一秒发送两段，也只是在常见情况下制造两次到达，不能证明分帧器覆盖了所有边界。

Unity 的 `TestNetwork.OnSendFenBaoButtonClick` 在主线程执行 `Thread.Sleep(1000)`，会让游戏和界面冻结一秒。

正确的核心测试不是“强迫 TCP 按我想的方式分包”，而是直接测试 `TcpFrameDecoder`：把相同的完整帧字节按不同切点调用 `Append`，验证输出始终一致。Socket 联调只作为第二层集成测试。

### 4.12 P2：其他值得顺手清理的问题

- `NetworkManager.Awake` 同步 `Connect`，网络不通时可能冻结 Unity 主线程；以后改为可取消的连接流程。
- IP 与端口不应硬编码在 `Awake` 和 `Program`，至少集中到配置字段或启动参数。
- 连接失败路径创建了 Socket，却没有统一释放；`Destroy(this)` 和随后 `Destroy(gameObject)` 也让生命周期更难跟踪。
- `CloseConnection` 未做 null/重复关闭保护，未清理流累积区，也没有把 `_clientSocket` 置空。
- 后台线程不应直接触碰 Unity 场景对象；网络层把事件或完整帧排到线程安全队列，Unity 主线程在 `Update` 中消费。
- `TestNetwork.Update` 每帧只处理一条消息。建议使用 `TryDequeue` 循环并设置每帧上限，例如最多 100 条，兼顾积压清理与帧率。
- 未知 `MessageTypeId` 应有明确策略：记录、丢弃该完整帧，必要时断开；不能静默无反馈。
- `BitConverter` 使用本机字节序。客户端和当前 Windows 服务端通常碰巧一致，但协议应明确固定字节序。整数、浮点和 bool 的线上表示都要集中定义；bool 建议固定为单字节 0/1。

## 六、推荐的分阶段还债顺序

### 阶段 0：冻结协议定义并止血

1. 在纸面和常量里明确当前帧格式：`int32 MessageTypeId + int32 PayloadLength + Payload`。
2. 明确 `PayloadLength` 不包含 8 字节头。
3. 固定字节序和最大 payload 长度。
4. 同时修复两端 `PlayerMsg` 的异常。
5. 两端实现 `SendAll`。
6. 所有入口对 null、非法长度和未知类型给出明确结果。

验收标准：`StringMsg`、`PlayerMsg` 都能多次序列化；头中长度永远等于实际 payload 长度；发送不会默默丢尾部。

### 阶段 1：先抽出纯分帧器，再接回网络

1. 新建 `TcpFrameDecoder`，不要引用 Socket、Task、MonoBehaviour 或业务消息类型。
2. 用“头不足 / 体不足 / 完整就产出并继续”的统一循环替代三套分支。
3. 增加最大长度、负长度、溢出检查。
4. 先写 EditMode 测试，再让客户端和服务端都调用同一份 decoder。
5. 删除两边重复的 `ProcessTCPOriginalBytes`。

验收标准：输入如何切块都得到完全相同、顺序一致的完整帧；残缺尾部会保留到下一次 `Append`。

### 阶段 2：消除空转和数据竞争

1. 客户端使用一个可等待的发送队列和一个发送者。
2. 收到的完整帧使用 `ConcurrentQueue` 交给 Unity 主线程。
3. 服务端删除中央 `Available` 轮询，为每个会话启动独立的阻塞接收循环。
4. 服务端使用线程安全活动会话表并在关闭时移除。
5. 每连接按顺序处理帧，不再每帧 `Task.Run`。

验收标准：空闲客户端和空闲服务端不再由自写循环持续占用 CPU；多客户端连接/断开/广播时不出现集合枚举异常和消息乱序。

### 阶段 3：整理消息编解码与共享代码

1. 消息类只处理 payload。
2. `MessageCodec` 统一写头、读头和校验长度。
3. 删除长度缓存、负数哨兵与公共 setter。
4. 使用 `MessageRegistry` 取代客户端、服务端各自的硬编码 switch。
5. 服务端链接同一份协议源码，删除复制版本。

验收标准：新增一种消息时，只需定义一次 TypeId、payload 编解码和注册；不需要同时手改两套头部与分帧代码。

### 阶段 4：完善生命周期和真正异步

1. 连接、发送、接收循环返回可观察的 `Task`。
2. 使用 `CancellationToken` 和统一连接状态机。
3. `Close/Dispose` 幂等并等待循环退出。
4. 把阻塞式 Socket 调用替换为 Async API。
5. 给连接、断开、协议错误、发送失败建立结构化日志和主线程事件。

验收标准：连接失败、远端关闭、本地主动关闭、重复关闭、退出 Unity Play Mode 都能干净结束，没有遗留任务和未观察异常。

## 七、必须补的自动测试

### 7.1 `TcpFrameDecoder` 单元测试

至少覆盖：

1. 只输入 0～7 个头字节，不产出帧。
2. 头刚好 8 字节但 payload 不完整，不产出帧。
3. 一条完整帧一次输入，产出一条。
4. 对同一条帧，在每一个可能字节位置切成两段，结果都正确。
5. 每次只输入 1 字节，最终正确恢复。
6. 两条、三条完整帧一次输入，按顺序全部产出。
7. 一条完整帧加下一条半帧，先产出第一条并保留第二条残片。
8. 零长度 payload。
9. 负长度、超过上限、加法溢出，返回协议错误而不是继续解析。
10. 随机生成切块边界，多轮验证输出与原始帧序列一致。

### 7.2 消息编解码测试

- `StringMsg`：空字符串、ASCII、中文、emoji、较长字符串往返一致。
- `PlayerMsg`：普通数据往返一致；嵌套 `PlayerData` 长度正确。
- 同一消息对象修改字段后再次序列化，头部长度跟随新 payload。
- 截断字符串、负字符串长度、超长字符串、截断嵌套对象都被拒绝。
- 头中 payload 长度与实际长度不一致时被拒绝。
- 未知 TypeId 走明确错误分支。

### 7.3 传输与生命周期集成测试

- 客户端连接后由服务端正常关闭，双方都清理状态。
- 客户端异常断开，服务端最终移除会话。
- 连续发送大量小消息，服务端保持顺序且没有丢帧。
- 多客户端同时连接、断开、广播，不出现字典并发异常。
- 主动关闭期间仍有阻塞 Receive/待发送项，所有循环最终结束。

## 八、为心跳、真正异步和 UDP 留出的边界

### 8.1 心跳

心跳不属于 `TcpFrameDecoder`，它是一种普通协议消息和连接监督策略：

- 定义 `Ping/Pong` 或单向 heartbeat 的 MessageTypeId。
- `ClientSession` 记录最近收到任何有效帧的时间。
- 单独的监督循环定时检查超时；不要在每帧 `Update` 创建任务。
- 超时只触发统一关闭流程。

这样以后添加心跳时，不改分帧算法和基础序列化。

### 8.2 真正异步

异步化应该只替换 Transport 层的等待方式：

- `Accept` → `AcceptAsync`
- `Receive` → `ReceiveAsync`
- `SendAll` 内的 `Send` → 循环 `SendAsync`
- 阻塞发送队列 → 可异步等待的队列

`TcpFrameDecoder`、`MessageCodec`、消息定义和业务分发接口保持不变。这就是现在先拆层的价值。

### 8.3 UDP

不要把 TCP 的分包/粘包解码器复用到 UDP：UDP 保留数据报边界，接收一次得到的是一个数据报（缓冲区过小时还可能截断），没有 TCP 字节流意义上的粘包处理。

可以复用：

- 消息类型 ID；
- payload 编解码；
- `MessageCodec` 的大部分逻辑；
- 消息注册与业务分发。

不能复用：

- `TcpFrameDecoder`；
- TCP 连接状态；
- TCP 的发送队列和重连语义。

以后新增独立 `UdpTransport`，把收到的数据报作为一个边界明确的输入进行头部和长度校验即可。可靠、顺序、重传等如果需要，应当作为 UDP 之上的新协议明确设计，不能假设 UDP 自带。

## 九、建议保留与暂缓的内容

### 现在保留

- 8 字节头的简单协议，足够当前学习阶段使用。
- 先使用同步阻塞 Socket + 明确的专用循环，前提是没有空转且能正确关闭。
- `StringMsg` 与 `PlayerMsg` 两种简单消息，作为测试协议的样本。

### 现在不要急着做

- 不必立刻引入复杂 ECS、响应式框架或大型网络库。
- 不必在正确性尚未稳定时先上对象池、零拷贝和极端性能优化。
- 不必为了未来 UDP 强行让 TCP/UDP 共用同一个分帧器。
- 不必现在就把心跳、重连、加密、压缩、请求响应 ID 全塞进协议头。

当 P0/P1 完成后，再考虑在头里增加协议版本、请求/序列号、压缩标记等字段。现在最重要的是让“协议定义、分帧、消息解析、传输、业务分发”各自只有一个清楚职责。

## 十、最终验收清单

完成本轮还债后，应能逐项回答“是”：

- [ ] 客户端和服务端使用同一份协议定义和消息源码。
- [ ] `PayloadLength` 的含义唯一且不含协议头。
- [ ] `PlayerMsg` 可以正常、重复地序列化和反序列化。
- [ ] 所有长度字段在分配和读取前都经过上下限与剩余长度校验。
- [ ] 任意 TCP 切块方式都能恢复相同的完整帧序列。
- [ ] 一次发送没有发完时会继续发送剩余字节。
- [ ] 空闲时没有自写 while 循环持续轮询。
- [ ] 普通 `Queue` / `Dictionary` 不再被多个线程无保护访问。
- [ ] 每个连接的消息默认按接收顺序处理。
- [ ] 服务端能发现正常断开并从活动会话表移除客户端。
- [ ] 主动关闭、远端关闭、连接失败和重复关闭都进入同一套幂等清理流程。
- [ ] Unity 主线程不再使用 `Thread.Sleep`，网络线程不直接操作场景对象。
- [ ] 核心分帧和编解码行为有自动测试，而不是只靠 Socket 手工实验。

完成这些以后，这套代码就具备继续学习心跳、真正异步和 UDP 的合理基础；新增功能会落到明确的一层，而不会再次堆回 `NetworkManager` 或 `ConnectClientSocket`。

## 四、固定术语与当前架构

### 4.1 TCP 与协议术语

| 术语 | 含义 | 当前代码 |
| --- | --- | --- |
| 接收块（receive chunk） | 一次 `Receive` 实际返回的那段字节，没有消息边界语义 | `buffer + bytesCount` |
| 流累积区（stream accumulator） | 保存尚未组成完整帧的 TCP 字节 | 当前为 `_originalBytesQueue` |
| 分帧（framing / frame decoding） | 根据协议头从连续 TCP 字节流恢复完整帧边界 | 当前为两端 `ProcessTCPOriginalBytes` |
| 粘包 / 分包 | 接收块边界与帧边界不一致的现象描述；不是两个独立协议 | 手工测试中的现象名称 |
| 协议头（header） | 当前为 `MessageTypeId + PayloadLength`，共 8 字节 | 两个 `int32` |
| 消息体（payload/body） | 协议头之后的编码数据 | 字符串或 `PlayerMsg` 字段数据 |
| 完整帧（frame） | 一条线上数据单元：`Header + Payload` | `_completeFrameQueue` 中的 `byte[]` |
| 消息（message） | 已解析、具有业务含义的对象 | `StringMsg`、`PlayerMsg` |
| 阻塞（blocking） | 线程等待 I/O 或信号时不继续执行，CPU 可去运行其他线程 | `Accept`、`Receive`、`GetConsumingEnumerable` |
| 局部引用快照 | 在锁内把共享 Socket 字段复制到局部变量；字段以后置空不会让局部变量变空，但对象仍可能被另一线程关闭 | `ClientSession.SendBytesToClient` / `Close` |
| 客户端会话（client session） | 服务端对一个已接受 TCP 连接的运行期管理对象，持有通信 Socket、接收循环、流累积区和关闭回调 | `ClientSession` |

“分帧”是准确的处理职责名称；“粘包/分包”是现象名称。统一规则应当是：头不足就等待，完整帧不足就等待，完整就产出并继续。无需为粘包和分包各设计一套解码器。

### 4.2 `CanBeBinarySerialize` 与完整帧的边界

当前代码必须按下面的真实语义理解：

```text
CanBeBinarySerialize<T>
└─ 只提供字段到字节、字节到字段的通用能力
   └─ 不会自动添加 MessageTypeId 和 PayloadLength

MsgBase<T> : CanBeBinarySerialize<T>
└─ 提供当前 8 字节消息头的读写方法
   ├─ StringMsg  → Header + string payload
   └─ PlayerMsg  → Header + player payload

PlayerData : CanBeBinarySerialize<PlayerData>
└─ 只有 PlayerData 自身字段字节，是嵌套 payload，不是完整网络帧
```

因此当前 `CanBeBinarySerialize.frameBytes`、`SerializeToFrameBytes()` 和 `GetFrameBytesLength()` 的命名存在语义偏差：当对象是 `PlayerData` 时，它们产生的不是 frame。这个命名问题不影响当前 P0 正确性，记入 P2-3；以后可以改成 `_encodedBytes`、`SerializeToBytes()`、`GetEncodedLength()`，或者拆成明确的 `ByteReader` / `ByteWriter`。

### 4.3 服务端、会话与 Socket 的层级

```text
TcpServer
├─ listener socket：字段 `_listenSocket`，只负责 Accept 新连接
└─ ConcurrentDictionary<int, ClientSession>：活动客户端会话表
   └─ ClientSession：一个已连接客户端的运行期对象
      ├─ communication socket：字段 `_connectClientSocket`
      ├─ 专属后台 Receive 线程
      ├─ 当前连接的流累积区
      └─ 关闭时通知 TcpServer 移除会话
```

监听 Socket 不叫 session。封装监听 Socket、会话表、广播和整体关闭的对象叫 `TcpServer`；只有一个已连接客户端对应的运行期对象才叫 `ClientSession`。

### 4.4 客户端和服务端的完整帧交付为何不同

```text
客户端
接收线程 → 分帧 → ConcurrentQueue<byte[]> _completeFrameQueue
                         ↓
Unity 主线程 Update → TryDequeue → 消息解析 → UI / 场景业务

服务端
ClientSession 接收线程 → 分帧 → ProcessReceivedFrame → 同步解析和处理
```

Unity 的 UI、场景对象和大多数 `UnityEngine` API 要在主线程操作，所以客户端需要并发队列跨线程交付。服务端当前没有 Unity 主线程约束，并且会话内同步处理能自然保留该客户端的消息顺序。两端应统一帧格式和消息协议，不必统一完整帧之后的线程交付模型。

当前同步服务端为每个已连接客户端创建一条专属接收线程。少量连接的学习项目可以这样做；连接数很大时线程栈、调度和上下文切换成本会成为问题，下一阶段用真正异步 I/O 替换等待方式。

## 五、Socket 返回值与三类关键异常

### 5.1 `Receive` 与关闭

| 结果 | 含义 |
| --- | --- |
| `Receive > 0` | 收到该数量的应用字节，继续累积和分帧。 |
| `Receive == 0` | 对端已正常关闭发送方向，并且此前字节已经读完；这不是异常，也不是对方发送了一个“0 字节”。 |
| `SocketException` | 网络、套接字状态或本地主动关闭中断了 I/O，需要结合当前关闭状态判断是否是预期现象。 |

永久结束连接时，当前目标流程仍是：尽力 `Shutdown(SocketShutdown.Both)`，然后 `Close()` 释放资源。关闭应最终做成幂等操作：多线程或多路径重复调用不会再次操作已释放对象。

### 5.2 当前项目中的异常

| 异常 | 定义 | 当前典型触发条件 | 当前处理与尚存缺口 |
| --- | --- | --- | --- |
| `InvalidDataException` | 输入字节不符合本项目协议，是协议内容错误，不是 Socket I/O 错误 | 字符串/嵌套对象长度为负数；基础字段或内容超过剩余字节；读取位置非法 | 客户端在 `TestNetwork.Update` 捕获、记录并关闭连接；服务端在 `ClientSession.ReceiveLoop` 捕获、记录并关闭会话。外层 `PayloadLength` 非法由分帧器直接记录和关闭，不通过该异常。 |
| `SocketException` | .NET Socket API 报告的网络或套接字操作错误 | `Connect` 失败；`Receive` / `Send` 遇到重置、断网或不可用连接；本地 `Shutdown/Close` 也可能唤醒阻塞调用 | 客户端接收循环会在主动关闭状态下静默退出；其他路径记录错误。服务端接收路径目前可能把整体关闭唤醒产生的 `SocketException` 也记录出来，留到 P1-1 统一状态机处理。 |

## 六、尚未处理的 8 个问题

这些问题按原报告计数保留，当前阶段不要求继续修改。

### 6.1 P1-1：异步任务与关闭生命周期不可管理

当前证据：

- `NetworkManager.ProgressSendFrameAsync`、`ProgressReceivedBytesAsync` 和 `TcpServer.AutoAcceptSocket` 仍是 `async void + Task.Run`。
- 调用者没有保存收发与 Accept 任务，无法 `await`、集中观察异常或等待干净退出。
- `_needConnectToServer`、`_isEnable` 是跨线程共享普通 `bool`，还没有取消令牌和明确状态机。
- `NetworkManager.CloseConnection` 没有把 `_clientSocket` 交换为 `null`，`Shutdown` 也没有覆盖重复关闭/已断开异常；连接失败路径没有统一释放刚创建的 Socket。
- `TcpServer` 在判空后又通过共享字段调用 `Accept`，关闭线程可能在两步之间把字段置空；这一竞态应在异步重构时用局部引用和统一关闭消除。
- `ClientSession` 保存了接收线程，但服务端关闭时没有等待线程退出。
- 客户端发送队列是单一发送者；服务端目前没有每会话发送队列。现在控制台基本是单一调用者，但加入心跳监督线程后，心跳和业务广播可能并发调用同一个会话的 `SendCompleteFrame`，两帧可能在多个部分发送之间交错。

后续处理点：把连接、Accept、Receive、Send 循环变成可观察 `Task`，使用 `CancellationToken` 和幂等 `CloseAsync/DisposeAsync`，并保证**每条 TCP 连接只有一个实际写 Socket 的发送循环**。这应当成为异步 TCP 学习阶段的首要任务，而不是现在继续还债。

### 6.2 P1-3：payload 长度仍可能陈旧

已经修好的部分：反序列化头部只读取网络中的 `MsgTypeID` 和 `PayloadLength`，不再调用本地 `SetPayloadLength()`，之前的 `PlayerMsg` 空引用路径已经消失。

尚存问题：

- `MsgBase` 仍用 `-999999999`、`-888888888` 作为未初始化哨兵，并公开可写的 `MsgTypeID` / `PayloadLength`。
- `StringMsg.GetFrameBytesLength()` 每次会更新 `cacheMsgBodyCount`，但 `TrySetMsgTypeID()` 只在 `PayloadLength` 仍是哨兵时调用 `SetPayloadLength()`。
- 因此同一个 `StringMsg` 第一次序列化后修改 `msg` 再序列化，数组长度可能按新字符串分配，而头部仍写旧的 `PayloadLength`。
- `PlayerMsg` 第一次序列化已正确，但序列化后再修改嵌套 `PlayerData` 也存在同类陈旧头长度风险。
- 反序列化器能防止越界，但尚未统一验证“消息解析结束时恰好消费完声明的 payload”；直接绕过网络分帧器调用反序列化时，也没有独立校验头长度与传入数组长度。

后续处理点：由 `MessageCodec.Encode` 先生成 payload，再用真实 `payload.Length` 一次性写头；消息类只读写 payload，删除缓存、哨兵和公共长度 setter。

### 6.3 P1-4：分帧器的复制和内存成本

两端算法目前功能方向正确，但仍然：

- 每次看头都调用 `_originalBytesQueue.ToArray()`，复制整个累积区。
- 接收块逐字节入队，完整帧逐字节出队。
- 客户端固定分配 1 MiB 接收数组；服务端每个 `ClientSession` 也固定分配 1 MiB。
- 客户端和服务端各自维护一份几乎相同的三分支分帧代码。

后续处理点：抽出纯 C# `TcpFrameDecoder`，使用连续缓冲区和读写下标，提供 `Append` / `TryReadFrame`。这不是当前功能阻塞项；可在异步 I/O 稳定后再优化，避免同时重写过多层。

### 6.4 P1-5：协议代码重复

下面五个类型仍在客户端和服务端各有一份：

- `CanBeBinarySerialize`
- `MsgBase`
- `StringMsg`
- `PlayerMsg`
- `PlayerData`

当前文件除 Unity `using` 等平台差异外基本一致，所以现阶段能够联调；问题是新增 `Ping/Pong`、字段、TypeId 或 Protobuf 时需要两边同步修改，漏改一边会直接形成协议漂移。

后续处理点：建立不引用 `UnityEngine` 的共享协议源码或独立类库，由两端引用同一份物理代码。若为了学习节奏先临时复制心跳消息，必须明确记录这是过渡实现，并在协议/Protobuf 阶段合并。

### 6.5 P1-6：缺少自动测试

当前手工入口仍包括：

- Unity `TestNetwork` 的分包、粘包、组合和非法字符串长度按钮。
- Unity 两个分包按钮在主线程调用 `Thread.Sleep(1000)`，会冻结一秒。
- 服务端 `Program` 的 `nianbao`、`fenbao`、`分包黏包` 和 `ff` 命令。

这些入口适合观察现象，但 `Send` 次数和 TCP 实际接收块边界没有一一对应关系，不能穷举证明分帧正确性。以后抽出 decoder 后，至少补：头不足、体不足、每个切点、逐字节输入、多帧、半帧尾部、0 payload、负数、超限和随机切块测试。

### 6.6 P2-1：同步连接、配置和客户端生命周期

- `NetworkManager.Awake` 同步调用 `Connect`；网络慢或不可达时可能阻塞 Unity 主线程。
- 客户端和服务端都硬编码 `192.168.0.196:8080`。
- 连接失败时创建的 Socket 没有走统一释放路径。
- `CloseConnection` 没有完整的重复关闭保护，也没有清理 `_originalBytesQueue` 或把 Socket 字段置空。
- `TestNetwork.Update` 直接假定 `NetworkManager.Instance` 始终存在；连接初始化失败时缺少显式空状态处理。

这些可自然并入异步连接、重连和配置学习，不需要现在单独返工。

### 6.7 P2-2：Unity 每帧只消费一条完整帧

`TestNetwork.Update` 当前使用一次 `if (TryDequeueReceivedFrame(...))`。低消息量没有问题；突发大量消息时，即使接收线程已经分好帧，主线程也只按每帧一条清理积压。

以后可改成带预算的循环，例如“本帧最多处理 N 条或最多处理 X 毫秒”，既消化积压又保护帧率。不要使用无限 `while` 把所有业务一次性压在一帧。

### 6.8 P2-3：协议定义、字节序、未知类型和命名

- 两端 `HeaderSize = 8`、`MaxPayloadLength = 256 KiB` 当前一致，但分别声明，不是单一真相来源。
- `BitConverter` 使用运行机器的本机字节序；线上协议尚未明确 Little Endian 或 Big Endian。
- TypeId `1000/1001` 分散在消息类型与两个 switch 中；未知 TypeId 当前被静默忽略。
- 协议没有版本字段；当前学习阶段可以不加，但跨版本部署前要设计兼容策略。
- `CanBeBinarySerialize` 的 `frameBytes` 命名会把纯 payload 的 `PlayerData` 也误称为完整帧。

后续在“协议 + Protobuf”阶段集中建立 `ProtocolConstants`、`MessageType`、固定字节序、未知类型策略和 codec/registry。

## 八、从当前基线继续学习的路线

### 8.1 第一阶段：真正异步 TCP

推荐先异步、再加正式心跳，因为异步阶段会先解决取消、关闭、任务等待和单一发送者；心跳可以直接建立在稳定的连接生命周期上。

原子学习顺序：

1. 让连接、Accept、Receive、Send 循环返回可观察 `Task`。
2. 引入 `CancellationTokenSource` 和明确连接状态。
3. `Accept` → `AcceptAsync`，`Receive` → `ReceiveAsync`。
4. 每连接建立唯一发送队列和唯一发送循环；`SendAsync` 仍须根据返回值循环到整帧完成。
5. 统一幂等关闭，并等待所有循环退出。
6. 保持当前帧格式和消息对象不变，先只替换 Transport 层等待方式。

### 8.2 第二阶段：心跳

心跳是普通协议消息和连接监督策略，不属于 TCP 分帧器：

- 定义 `Ping/Pong` 或单向 heartbeat 消息。
- 连接收到任意有效帧时更新最后活跃时间。
- 一个监督循环定时检查超时，不要在每帧 `Update` 创建任务。
- 超时只调用统一关闭流程。
- 心跳和业务消息必须进入同一个发送队列，不能让多个线程直接并发写同一 Socket。

如果为了学习顺序决定先在当前同步版做心跳，服务端至少要先保证整个 `SendCompleteFrame` 对每个会话串行执行；否则心跳线程和控制台/业务线程可能交错写帧。

### 8.3 后续：FTP、HTTP、UnityWebRequest、协议、Protobuf

这些主题处在不同层，不要全塞进当前 TCP 类：

| 主题 | 所属层与本项目关系 |
| --- | --- |
| FTP | 独立的应用层协议学习；可练习连接、命令/数据通道和文件传输，但不应混进游戏长连接的 `ClientSession` 分帧逻辑。 |
| HTTP | 请求/响应式应用协议；服务端接口与当前 TCP 长连接可以并存，各自拥有 Transport/Endpoint。 |
| UnityWebRequest | Unity 侧访问 HTTP/HTTPS 的客户端 API；它不替代当前自定义 TCP 分帧器。 |
| 协议设计 | 集中定义消息类型、版本、错误策略、共享源码和注册表，解决 P1-3/P1-5/P2-3。 |
| Protobuf | 解决 payload 的结构化序列化和兼容演进；TCP 仍是字节流，外层仍需要明确帧边界或采用约定的长度前缀。 |

面向以后小游戏服务端底座，应持续保持四层分离：传输、TCP 分帧、消息序列化、业务/会话。当前 Unity 客户端代码不应渗入 .NET 服务端；到真正接入目标平台时，再单独核验该平台当时支持的网络 API 和部署约束。

### 8.4 UDP 不在当前计划中

本阶段不继续 UDP，也不需要为了“将来可能用 UDP”改造当前 TCP 分帧器。UDP 数据报与 TCP 字节流边界不同；以后如果目标改变，应新增独立 `UdpTransport`，而不是复用 `TcpFrameDecoder`。本报告后续路线不再把 UDP 列为下一步。
